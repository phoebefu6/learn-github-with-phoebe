# Presenter notes - Session 2 · Your first repo + Markdown

## Preflight (10 min before)
- Open courses/02-first-repo-markdown.html, click Expand all, set Projector zoom: on.
- Test the mdpreview widget once: type a # heading and a **bold** word, confirm the right pane renders live.
- Have a throwaway demo repo ready under your own account (so you can show the + button flow without touching your real handbook).
- Log in to github.com in a second tab; check the pencil editor and Preview tab load fast on the room wifi.

## Run of show
- 0-3 · Welcome. One line: "Last week you got the map. Today you become a homeowner." Show the callout win box - the promise is a real repo by minute 40.
- 3-7 · Part 1, create a repo. Walk the SVG top to bottom: name, description, visibility, README checkbox. Do the naming bit out loud: say "TeamHandbook_v2_2026" in a flat voice, then "team-handbook". The room laughs, the lesson lands.
- 7-11 · Part 2, README. The four questions (what, who, how, owner). Demo the pencil → Preview → Commit loop once on your demo repo. Tell the awesome-lists story with the star counts - a README-only repo out-starring nearly every app on earth.
- 11-18 · Part 3, Markdown. Walk the annotated SVG left panel to right panel, then the dialect table fast - do NOT read every row. Land hard on checkboxes (- [ ]) and drag-drop images. Point at the self-study card for the social extras.
- 18-24 · Apply-along 1. Everyone creates team-handbook and writes the 4-line front door. Circulate; the usual snag is missing the Add a README checkbox - they land on a scary setup page. Fix: delete nothing, just create the README via Add file.
- 24-34 · Apply-along 2. The playground. Give them the shopping list (title, bold intro, section, 3-item list with a link, Day/Focus table), then go quiet and let them type. At ~31 min call the move: copy playground → real README → Preview → commit.
- 34-40 · Apply-along 3. Upgrade the Session 1 profile README with today's markdown. End with everyone visiting their own profile - hold the beat, let them admire.
- 40-45 · Quiz (90 seconds, read answers aloud) then Q&A. Point at homework + cheat sheet on the way out.

## Never cut
- The commit-message-as-note-to-future-you framing. "clarify expense policy" vs "update" is the single habit that survives this course.
- The copy-playground-into-real-README moment. Practice becoming production, live, is the emotional payoff of the whole session.

## Cut if long
- The public vs private table (self-study card carries it; just say "when unsure, private").
- The commit-messages table walkthrough - say the rule, skip the rows.
- Apply-along 3 step 3 (the table-or-quote flourish); heading + intro + list is enough.

## Expected questions
- "Can I rename a repo later?" Yes, Settings → rename; GitHub even redirects the old URL. But links in people's chats age badly, so name well once.
- "Public repo - can strangers edit my handbook?" No. Anyone can READ; only people you invite can change anything. Proposals from strangers still need your approval (Session 4).
- "Why Markdown instead of Word or Google Docs?" It never breaks, every change is tracked line by line with a name on it, and it turns into a live website in Session 5. Docs are fine for drafts; this is for the source of truth.
- "Can I make just one file private?" No - visibility is per-repo. Sensitive page? It goes in a separate private repo.
- "Is there a WYSIWYG editor so I can skip the symbols?" The Preview tab is the safety net, and github.dev exists, but 10 symbols cover 95% of writing - the playground just proved it.
